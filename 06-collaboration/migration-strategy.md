# Migration Strategy

> 레거시 디자인 및 컴포넌트에서 Carmore Design System으로의 단계별 마이그레이션

---

## 1. 개요

기존 레거시 UI를 새로운 디자인 시스템으로 전환하는 과정은 **점진적이고 안전하게** 진행합니다. 한 번에 전체를 교체하는 빅뱅 방식이 아닌, 단계별 마이그레이션으로 서비스 안정성을 유지합니다.

---

## 2. 마이그레이션 원칙

| 원칙 | 설명 |
|------|------|
| **점진적 전환** | 한 번에 전체 교체하지 않고, 페이지/컴포넌트 단위로 전환 |
| **공존 허용** | 전환 기간 동안 레거시와 새 시스템이 공존할 수 있음 |
| **하위 호환** | 새 토큰/컴포넌트는 기존 인터페이스를 깨뜨리지 않음 |
| **측정 가능** | 전환 진행률을 추적할 수 있는 기준 수립 |
| **롤백 가능** | 문제 발생 시 이전 상태로 안전하게 복구 가능 |

---

## 3. 4단계 마이그레이션 로드맵

```
Phase 1          Phase 2          Phase 3          Phase 4
Foundation   →   Core Components  →  Complex        →  Patterns
토큰 구축         핵심 컴포넌트 전환     복합 컴포넌트 전환     패턴/가이드 완성
(2주)            (3주)              (3주)              (2주)
```

---

### Phase 1: Foundation (토큰 구축) — 2주

**목표**: 디자인 토큰을 코드와 Figma에 배포하고, 레거시 색상/간격/폰트의 매핑 테이블 작성

#### 작업 목록

```
Week 1:
□ 레거시 색상 값 감사 (Audit) — 사용 중인 모든 색상 값 수집
□ 레거시 색상 → 새 토큰 매핑 테이블 작성
□ CSS Custom Properties / Swift / Compose 토큰 패키지 생성
□ Figma Foundation 라이브러리 구축 및 퍼블리시

Week 2:
□ 레거시 타이포그래피 → 새 Type Scale 매핑
□ 레거시 간격 값 → $dimension 토큰 매핑
□ 토큰 패키지 배포 (npm / CocoaPods / Gradle)
□ 개발자 온보딩: 토큰 사용법 워크숍
```

#### 매핑 테이블 예시

| 레거시 값 | 새 토큰 | 비고 |
|-----------|---------|------|
| `#1a73e8` (기존 primary) | `$color.bg.brand` (#0077ED) | 약간의 색상 차이 |
| `#333333` (기존 text) | `$color.fg.neutral` (#1F2430) | |
| `#999999` (기존 secondary text) | `$color.fg.neutral-weak` (#6B7280) | |
| `#f5f5f5` (기존 background) | `$color.bg.neutral-weak` (#F7F8FA) | |
| `font-size: 14px` | `$typography.body2` (t7Regular) | |
| `padding: 15px` | `$dimension.x4` (16px) | 1px 차이 허용 |
| `border-radius: 4px` | `$radius.xs` (4px) | 동일 |

#### Phase 1 완료 기준

```
✅ 토큰 패키지가 모든 플랫폼에 배포됨
✅ Figma Foundation 라이브러리가 활성화됨
✅ 레거시 → 토큰 매핑 테이블 100% 작성됨
✅ 개발팀 온보딩 완료
```

---

### Phase 2: Core Components (핵심 컴포넌트 전환) — 3주

**목표**: 가장 사용 빈도 높은 Button, TextField, Card 3개 컴포넌트를 새 디자인 시스템으로 전환

#### 선정 기준

| 컴포넌트 | 사용 빈도 | 전환 우선순위 | 이유 |
|----------|-----------|-------------|------|
| **Button** | 최고 | 1순위 | 모든 화면에 존재, 영향 범위 최대 |
| **TextField** | 높음 | 2순위 | 검색/폼 핵심 요소 |
| **Card** | 높음 | 3순위 | 리스트 화면의 핵심 |

#### 작업 플로우 (컴포넌트 당)

```
1. 현황 감사 (1일)
   └─ 레거시 컴포넌트의 모든 사용처와 variant 파악
   └─ 새 컴포넌트와의 차이점 목록 작성

2. Figma 컴포넌트 구축 (2일)
   └─ Foundation 토큰 기반으로 컴포넌트 제작
   └─ 모든 variant × state 제작
   └─ 퍼블리시

3. 코드 컴포넌트 구현 (3~5일)
   └─ Web: React/Vue 컴포넌트
   └─ iOS: UIKit/SwiftUI 컴포넌트
   └─ Android: Compose 컴포넌트
   └─ 토큰 참조로 스타일 적용
   └─ 접근성 요구사항 구현

4. 전환 적용 (2~3일)
   └─ 영향 범위 적은 페이지부터 순차 교체
   └─ AB 테스트 또는 Feature Flag로 점진적 롤아웃
   └─ QA: 시각적 회귀 테스트

5. 레거시 제거 (1일)
   └─ 전체 전환 완료 후 레거시 컴포넌트 deprecated 처리
   └─ 일정 기간 후 완전 삭제
```

#### Phase 2 완료 기준

```
✅ Button, TextField, Card가 새 디자인 시스템 기반으로 전환됨
✅ 레거시 컴포넌트 사용처가 0 (또는 deprecated 예정)
✅ 시각적 회귀 없음 (스크린샷 비교 통과)
✅ 접근성 요구사항 충족 (WCAG AA)
```

---

### Phase 3: Complex Components (복합 컴포넌트 전환) — 3주

**목표**: BottomSheet, Dialog, FilterChip, ComparisonTable 전환

| 컴포넌트 | 난이도 | 전환 순서 | 비고 |
|----------|--------|-----------|------|
| **FilterChip** | 중 | 1순위 | 검색 결과 필터에 즉시 적용 |
| **Dialog** | 중 | 2순위 | 예약 확인/취소에 영향 |
| **BottomSheet** | 상 | 3순위 | 플랫폼별 네이티브 대응 복잡 |
| **ComparisonTable** | 상 | 4순위 | 반응형 대응 복잡 |

#### Phase 3 완료 기준

```
✅ 모든 7개 핵심 컴포넌트가 새 시스템 기반으로 전환됨
✅ 플랫폼별 네이티브 대응 완료
✅ 레거시 컴포넌트 전체 deprecated
```

---

### Phase 4: Patterns & Polish (패턴 및 마무리) — 2주

**목표**: UI/UX 패턴 가이드 완성, 플랫폼 가이드 점검, 잔여 레거시 정리

#### 작업 목록

```
Week 1:
□ 검색 퍼널 → 새 컴포넌트 기반 플로우 재구성
□ 예약 플로우 → 폼 패턴 + 에러 처리 통합
□ Empty State / Loading State 패턴 전체 적용
□ 플랫폼별 가이드 최종 점검

Week 2:
□ 레거시 코드 완전 제거
□ 디자인 시스템 문서 최종 리뷰
□ Figma 라이브러리 정리 (미사용 컴포넌트 삭제)
□ 팀 전체 리트로스펙티브
```

#### Phase 4 완료 기준

```
✅ 레거시 코드/디자인이 완전히 제거됨
✅ 모든 화면이 디자인 시스템 토큰과 컴포넌트를 사용함
✅ 디자인 시스템 문서가 최신 상태임
✅ 팀 리트로 완료 및 개선점 반영
```

## 4. 진행률 추적

### 메트릭

| 지표 | 측정 방법 | 목표 |
|------|-----------|------|
| **토큰 커버리지** | 하드코딩 값 대비 토큰 사용 비율 | 95% 이상 |
| **컴포넌트 전환율** | 레거시 vs 새 컴포넌트 인스턴스 수 | 100% 전환 |
| **Figma 토큰 적용률** | Variable 사용 vs Detached 스타일 | 100% |
| **접근성 점수** | WCAG AA 충족 비율 | 100% |

## 5. 리스크 및 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 레거시와 새 시스템 공존 기간이 길어짐 | 유지보수 비용 증가 | Phase별 마감 기한 엄수, Feature Flag 활용 |
| 디자인-개발 간 토큰 불일치 | 시각적 불일치 | 자동화된 토큰 동기화 파이프라인 구축 |
| 팀원 온보딩 지연 | 전환 속도 저하 | Phase 1에 워크숍 포함, 페어 작업 |
| 사용자에게 갑작스러운 UI 변경 | UX 혼란 | AB 테스트 + 점진적 롤아웃 |
| 접근성 회귀 | 법적/UX 리스크 | 자동 접근성 테스트 CI 포함 |

---

## 6. 마이그레이션 후 유지보수

마이그레이션 완료 후 디자인 시스템은 **지속적으로 운영**됩니다.

| 활동 | 주기 | 담당 |
|------|------|------|
| 토큰 값 검토 | 분기 | 디자인 시스템 팀 |
| 새 컴포넌트 요청 리뷰 | 수시 | 디자인 시스템 팀 + PO |
| Figma 라이브러리 정리 | 월 1회 | 디자이너 |
| 접근성 감사 | 분기 | QA + 디자이너 |
| 문서 최신화 | 변경 시 | 변경 요청자 |
| 팀 리트로 | 분기 | 전체 팀 |
